# Lunedoc

Lunedoc is a privacy-first PDF tools SaaS. This is the monorepo for the web app, the backend API, and the design-canvas prototype.

For the long-form context (decisions, roadmap, current phase status, open questions), read **[`docs/project-status.md`](docs/project-status.md)** first. This README only covers running the project locally.

---

## Layout

```
lune-doc/                       ← pnpm workspace
├── apps/
│   └── web/                    ← @lunedoc/web · Vite + React 19 + TS · port 5173 — ONLY frontend app
├── packages/
│   ├── ui/                     ← @lunedoc/ui · design tokens + Header/Footer/ToolCard/DropZone/…
│   ├── i18n/                   ← @lunedoc/i18n · 346-key EN/TR/ES JSON tables + useI18n hook
│   ├── api/                    ← @lunedoc/api · typed HTTP client used by apps/web
│   └── tools/                  ← @lunedoc/tools · 8 tool widgets (Merge + Split wired to the real API)
├── services/
│   └── api/                    ← Backend FastAPI service · port 8000 · uv-managed
├── docs/                       ← Project docs + design-canvas prototype source files
├── index.html                  ← Prototype entrypoint (untouched; design source of truth)
└── pnpm-workspace.yaml
```

There is no `apps/marketing/` — the SEO landing routes were consolidated into `apps/web` on 2026-05-05. Static HTML for all 27 routes is generated by a small post-build prerender step inside `apps/web` (see below).

## Prerequisites

Install once on your machine:

| Tool | Min version | Why |
|---|---|---|
| Node | 22.12 | `apps/web` build + `pnpm` |
| pnpm | 9 | Workspace package manager |
| Python | 3.12 | Backend service runtime |
| [`uv`](https://docs.astral.sh/uv/) | 0.11 | Python deps + venv for `services/api/` |
| PostgreSQL | 16 | Backend database (host service — **no Docker**) |
| Redis | 7 | Celery broker (host service — **no Docker**) |
| `libmagic1` | — | MIME sniffing in the backend |
| `ghostscript` | 10+ | PDF compression (`/jobs/compress`). Falls back to PyMuPDF if absent — modest reduction only. |
| `libreoffice` | 24+ | Office round-trips for `/jobs/convert` (PDF↔DOCX/PPTX, DOCX→PDF). Image directions and PDF→JPG/PNG run on PyMuPDF and don't need it. |
| `tesseract-ocr` + lang packs | 5+ | OCR (`/jobs/ocr`). **Required — no fallback engine.** Install one language pack per UI locale (`-eng`, `-tur`, `-spa` for the EN/TR/ES surfaces). Free tier capped at 20 pages per file. |

On Xubuntu / Ubuntu:

```bash
# Node + pnpm: https://nodejs.org and https://pnpm.io
curl -LsSf https://astral.sh/uv/install.sh | sh        # uv
sudo apt install -y postgresql redis-server libmagic1 ghostscript libreoffice \
                    tesseract-ocr tesseract-ocr-eng tesseract-ocr-tur tesseract-ocr-spa
```

### One-time Postgres setup

```bash
sudo -u postgres psql <<'SQL'
CREATE ROLE lunedoc WITH LOGIN PASSWORD 'lunedoc_dev_password';
CREATE DATABASE lunedoc_dev  OWNER lunedoc;
CREATE DATABASE lunedoc_test OWNER lunedoc;
SQL
```

If the role already exists with a different password, `ALTER ROLE lunedoc WITH PASSWORD 'lunedoc_dev_password';`. See `services/api/README.md` for the full backend setup including Postgres role troubleshooting.

### One-time backend bootstrap

```bash
cd services/api
uv sync                                      # creates .venv + installs deps
cp .env.example .env                          # then edit OWNER_TOKEN_PEPPER
python3 -c "import secrets; print(secrets.token_urlsafe(32))"  # paste into .env
```

## Run the project locally

You need **three terminals** (backend API + Celery worker + frontend dev server). Postgres + Redis run as background host services and don't need a dedicated terminal.

### Terminal 1 — Backend API

> ⚠️ Run backend commands **from `services/api/`**, never from the repo root.

```bash
cd services/api
source .venv/bin/activate
uv sync                                       # idempotent
uv run alembic upgrade head                   # apply migrations to lunedoc_dev
uv run uvicorn lunedoc_api.main:app --port 8000 --reload
```

### Terminal 2 — Celery worker

The worker handles Merge + Split jobs. Skip this terminal if you only want to test file uploads.

```bash
cd services/api
source .venv/bin/activate
uv run celery -A lunedoc_api.workers.celery_app worker --loglevel=info
```

The TTL sweeper (deletes files past their 1-hour expiry) lives in Celery beat. Start it as a fourth process if you want sweeps in dev:

```bash
cd services/api
source .venv/bin/activate
uv run celery -A lunedoc_api.workers.celery_app beat --loglevel=info
```

### Terminal 3 — Frontend

```bash
cd apps/web
cp .env.example .env 2>/dev/null || true      # already points at localhost:8000
pnpm install                                  # idempotent
pnpm dev                                      # http://localhost:5173
```

### Health check

In any terminal:

```bash
curl http://localhost:8000/api/v1/healthz
```

Expected:

```json
{"status":"ok","version":"0.1.0","db":"ok","redis":"ok"}
```

If `db` is `unreachable`, Postgres isn't running or the role/database isn't set up — see the one-time setup above. If `redis` is `unreachable`, run `sudo systemctl start redis-server`.

## What runs where

| URL | Served by |
|---|---|
| `http://localhost:5173/` | apps/web (Vite dev server, React Router SPA) |
| `http://localhost:5173/merge-pdf`, `/split-pdf`, `/tr/merge-pdf`, `/es/split-pdf`, … | apps/web — same SPA, locale-aware routes |
| `http://localhost:8000/api/v1/*` | services/api (FastAPI) |
| `http://localhost:8000/docs` | services/api OpenAPI explorer |
| `http://localhost:8765/` | Prototype design canvas (run separately with `pnpm prototype:serve`) |

The prototype at `index.html` is a frozen design source of truth. It does **not** need to run for normal development; only when iterating on the visual design itself.

## Build / test

```bash
# Backend test suite (must run inside services/api)
cd services/api && source .venv/bin/activate
uv run pytest -v

# Frontend production build (TS + Vite + 27-route SEO prerender)
pnpm --filter @lunedoc/web build

# Faster iteration without prerender (TS + Vite only)
pnpm --filter @lunedoc/web build:vite-only
```

Backend tests need both the dev DB and the test DB migrated:

```bash
cd services/api && source .venv/bin/activate
uv run alembic upgrade head
uv run alembic -x db=test upgrade head
uv run pytest -v
```

## SEO landing pages

`apps/web` ships with a post-build prerender step (`apps/web/scripts/prerender.ts`, run via `tsx` after `vite build`) that writes per-route static HTML for SEO:

- 1 home (EN at `/`) + 2 localized homes (`/tr/`, `/es/`)
- 8 tool landing pages × 3 locales = 24 tool routes
- **27 routes** total under `apps/web/dist/`

Each generated `index.html` carries a per-route `<title>`, `<meta name="description">`, `<link rel="canonical">`, `hreflang × 4` (en/tr/es/x-default), and 4 JSON-LD blocks (SoftwareApplication / FAQPage / HowTo / BreadcrumbList) with `<html lang="...">` matching the locale.

The body is hydrated client-side from the React SPA. JS-disabled crawlers see only the head metadata; modern Google/Bing execute JS and see the full body.

Per-tool data tables (FAQ, HowTo, hero copy, honesty clauses) live at `apps/web/src/data/<tool>-pdf.ts`.

## Common pitfalls

- **Don't run backend commands from the repo root.** `services/api/` has its own `.venv/`, `pyproject.toml`, and `alembic.ini`. Running `alembic` or `uvicorn` from anywhere else won't find them.
- **`.env` files are gitignored.** Each new clone needs `apps/web/.env` and `services/api/.env` recreated from the `.env.example` siblings.
- **No Docker.** Postgres + Redis must be installed as system services, not containers, per project decision D11.
- **`apps/marketing/` no longer exists.** If you have an old branch/clone, run `git pull` and `pnpm install` to clean stale workspace state.
- **The Celery worker isn't optional for Merge/Split.** Without it, `POST /api/v1/jobs/merge` returns `202` but the job stays `queued` forever. Health check still says ok because Redis is up — only the worker is missing.
